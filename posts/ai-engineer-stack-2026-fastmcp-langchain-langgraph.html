Why these three, and why now.

A TikTok from Stekunda Henry recently made the rounds: 3 Python libraries you should learn to get hired as an AI Engineer in 2026. The three were FastMCP, LangChain, and LangGraph. The take was right, but the format was a 60-second clip. It deserves a longer treatment.

These three libraries are not competitors. They occupy different layers of the same stack, and once you understand what layer each one owns, the question stops being "which one should I learn?" and becomes "in what order, and how deeply?" This post answers that question with code, context, and a clear learning path.

The short version: FastMCP gives your AI agent secure, standardised access to the outside world. LangChain is the high-level framework for building LLM-powered applications. LangGraph is the stateful runtime underneath LangChain that makes complex, multi-step agents reliable in production. Together, they cover every layer from protocol to orchestration.

Current versions (June 2026): FastMCP 3.1 (GA: Feb 18, 2026) · LangChain 1.0 (GA: Oct 22, 2025) · LangGraph 1.2 (released: May 11, 2026)

FastMCP: The USB-C for AI Tools.

To understand FastMCP, you first need to understand the problem it solves.

Before November 2024, connecting an AI agent to an external system — a database, a Slack workspace, a REST API — meant writing a custom integration for each agent-tool pair. You had N agents and M tools, so you were writing up to N×M glue layers. Every framework had its own tool format. Nothing was portable.

On November 25, 2024, Anthropic released the Model Context Protocol (MCP) — an open JSON-RPC 2.0 specification that standardises how AI agents discover and call external capabilities. The analogy that stuck: USB-C for AI. One standard connector, everything plugs in.

MCP SDK downloads grew roughly 970× in 18 months. By Q1 2026, every major AI vendor — Anthropic, OpenAI, Google, Microsoft — offers native MCP support. The protocol crossed 2,000 community server implementations and became the default way the industry thinks about tool connectivity.

FastMCP is the Python implementation that makes building MCP servers trivial. It is the fast, Pythonic way to create MCP servers and clients, and is downloaded over a million times a day. Some version of FastMCP powers approximately 70% of MCP servers across all languages.

The three pillars of MCP

Tools

Functions an AI model can call. Think: search a database, send an email, run a calculation. The model decides when to call them based on the user's request.

Resources

Read-only data the model can access. Think: a file, a database record, a URL's content. Resources provide context without executing actions.

Prompts

Reusable prompt templates exposed by the server. Clients can list and use them to standardise how the model is instructed for common tasks.

FastMCP in code

This is the entire code for a working MCP server that exposes three tools:

from fastmcp import FastMCP import httpx # 1. Create the server mcp = FastMCP("My AI Assistant") # 2. Expose a tool — schema is auto-generated from the function signature @mcp.tool def add(a: int, b: int) -> int: """Add two numbers together.""" return a + b # 3. Expose a resource @mcp.resource("config://settings") def get_settings() -> str: """Return application config as JSON.""" return '{"env": "production", "version": "3.1"}' # 4. Expose a prompt template @mcp.prompt def summarize(text: str) -> str: return f"Summarize the following in 3 bullet points:\n\n{text}" # 5. Run the server if __name__ == "__main__": mcp.run()

That's it. FastMCP automatically generates the JSON Schema from your type hints, handles transport negotiation, manages authentication, and makes the server discoverable by any MCP client — Claude Desktop, VS Code Copilot, a LangGraph agent, or your own custom client.

The decorator pattern will feel familiar if you've used FastAPI. That's intentional. FastMCP was designed to have the same ergonomics for MCP that FastAPI has for REST.

What FastMCP 3.0 added

The 3.0 release (GA February 18, 2026) introduced the three-layer architecture that makes FastMCP composable at scale:

Providers

Where components come from. A provider can be decorated Python functions, a directory of files, a remote MCP server, or an OpenAPI spec. Swap providers without rewriting tools.

Transforms

Middleware for the component pipeline. Intercept and modify the flow of tools, resources, and prompts between providers and clients. Use them for auth checks, logging, or schema transformation.

Client CLI

Built-in commands: fastmcp list, fastmcp call, fastmcp discover. Generate a typed CLI script from any MCP server with fastmcp generate-cli.

FastMCP also ships with tool versioning (@mcp.tool(version="2.0")), OpenTelemetry instrumentation out of the box, and granular authorization scopes — production-ready by default, not as an afterthought.

Why FastMCP matters for your career: MCP is now the default tool integration standard. Every enterprise AI project you'll work on in 2026 needs to connect agents to internal APIs, databases, and SaaS tools. FastMCP is how you build those bridges. It's a skill that compounds — one MCP server you build becomes reusable across every agent framework that speaks the protocol.

LangChain: The agent engineering platform.

LangChain launched in October 2022 as a toolkit for chaining LLM calls together. By October 2025, it reached 1.0 — a stable, production-committed release built around one insight: the hard part of building with LLMs isn't the model call. It's the orchestration around it.

LangChain 1.0 describes itself as the "agent engineering platform." 35% of Fortune 500 companies use it, primarily for conversational agents. 84.7% of its community codes in Python. It's the framework you reach for when you need to connect an LLM to the real world and don't want to rebuild every integration from scratch.

The seven core components

Chat Models

A unified interface to 1,000+ LLM providers: OpenAI, Anthropic, Google, Cohere, Ollama (local), and more. Swap models without rewriting your application logic.

Prompt Templates

Parameterised prompt structures that separate your instructions from your data. Composable, testable, and reusable across chains and agents.

Chains (LCEL)

LangChain Expression Language — the composition layer. Pipe components together with | syntax. Chains are lazy, streaming-aware, and parallelism-friendly.

Tools & Toolkits

Python functions the agent can call. Over 300 pre-built integrations: web search, SQL databases, REST APIs, vector stores, calculators, and shell commands.

Memory

Persistent context across turns. Short-term (in-context), long-term (vector store or database), and summary memory. The agent remembers your name next session.

Retrievers

The RAG layer. Retrieval-Augmented Generation — pull relevant documents from a vector store at query time to ground the model's answers in your data.

Agents

The decision loop. The model reasons about which tool to call, calls it, observes the result, and continues until the task is complete. Powered by LangGraph at the runtime layer.

LangChain in code: a RAG pipeline

This is a minimal Retrieval-Augmented Generation chain — the pattern behind most production knowledge-base assistants:

from langchain_openai import ChatOpenAI, OpenAIEmbeddings from langchain_community.vectorstores import FAISS from langchain_core.prompts import ChatPromptTemplate from langchain_core.output_parsers import StrOutputParser from langchain_core.runnables import RunnablePassthrough # 1. Embed and index your documents docs = ["FastMCP powers 70% of MCP servers.", "LangGraph 1.0 shipped Oct 2025."] vectorstore = FAISS.from_texts(docs, OpenAIEmbeddings()) retriever = vectorstore.as_retriever() # 2. Define a prompt with context injection prompt = ChatPromptTemplate.from_template(""" Answer the question using only the context below. Context: {context} Question: {question} """) # 3. Build the chain with LCEL pipe syntax chain = ( {"context": retriever, "question": RunnablePassthrough()} | prompt | ChatOpenAI(model="gpt-4o") | StrOutputParser() ) # 4. Invoke it result = chain.invoke("When did LangGraph 1.0 ship?") print(result) # "LangGraph 1.0 shipped in October 2025."

The | pipe syntax is LCEL — LangChain Expression Language. Every step is a Runnable with a consistent interface. The chain streams by default, handles errors uniformly, and parallelises automatically when branches are independent.

LangChain agents with tools

from langchain_openai import ChatOpenAI from langchain.agents import create_react_agent, AgentExecutor from langchain_community.tools.tavily_search import TavilySearchResults from langchain_core.prompts import ChatPromptTemplate # Tools the agent can use tools = [TavilySearchResults(max_results=3)] llm = ChatOpenAI(model="gpt-4o") prompt = ChatPromptTemplate.from_template(""" You are a helpful research assistant. Use tools to answer questions accurately. {input} {agent_scratchpad} """) # Create a ReAct agent: Reason → Act → Observe loop agent = create_react_agent(llm, tools, prompt) executor = AgentExecutor(agent=agent, tools=tools, verbose=True) result = executor.invoke({"input": "What is FastMCP 3.0 and when was it released?"}) print(result["output"])

What changed in LangChain 1.0

The 1.0 release committed to three things: no breaking changes until 2.0, LangGraph as the underlying agent runtime, and a middleware system for production concerns (human-in-the-loop, PII redaction, rate limiting). LangChain agents now run on LangGraph under the hood. The create_agent abstraction wraps a full LangGraph state machine.

This is the key architectural point: LangChain is now a high-level API on top of LangGraph. If you want the convenient batteries-included experience, use LangChain. If you need full control over the state machine, drop to LangGraph directly.

LangGraph: Stateful agents that don't break.

LangChain's simple agent loop — Reason → Act → Observe → Repeat — is fine for quick tasks. It breaks for anything long-running: multi-step research, approval workflows, agents that need to pause for human input, systems that must survive server restarts mid-task.

LangGraph is the answer. Released as 1.0 in October 2025 alongside LangChain 1.0, and updated to 1.2 in May 2026, it is a "low-level orchestration framework for building, managing, and deploying long-running, stateful agents." It draws inspiration from Google's Pregel and Apache Beam — the same graph-based computation models that power distributed data pipelines at scale.

Companies running LangGraph in production: Klarna (85M users, 80% reduction in resolution time), Uber (large-scale code migration agents), LinkedIn (SQL Bot — natural language to SQL for 20,000+ employees), Coinbase, BlackRock, JPMorgan, Cisco, and Replit. Over 400 companies run LangGraph Platform deployments.

The core model: StateGraph

LangGraph models an agent as a directed graph. Every node is a computation step. Every edge is a transition between steps. The entire graph shares a typed state object that flows through every node and is automatically checkpointed at each step.

State

A TypedDict that holds everything the agent knows: conversation history, intermediate results, tool outputs, error counts. Every node reads and writes this shared object.

Nodes

Python functions (sync or async) that take the current state, do work — call an LLM, invoke a tool, run code — and return an updated state. Nodes are your agent's steps.

Edges

Connections between nodes. Static edges always go the same way. Conditional edges use a routing function: read the state, return the name of the next node. This is how you build branching logic and loops.

Checkpointing

Every state transition is saved to a pluggable saver (memory, SQLite, Postgres). The graph can pause, resume, time-travel to any previous state, and survive server restarts.

LangGraph in code: a research agent

from typing import TypedDict, Annotated from langgraph.graph import StateGraph, END from langgraph.checkpoint.memory import MemorySaver from langchain_openai import ChatOpenAI # 1. Define the state schema class AgentState(TypedDict): messages: Annotated[list, "conversation history"] research: str draft: str approved: bool llm = ChatOpenAI(model="gpt-4o") # 2. Define nodes — each takes state, returns partial state update def research_node(state: AgentState) -> dict: query = state["messages"][-1] result = llm.invoke(f"Research this topic thoroughly: {query}") return {"research": result.content} def write_node(state: AgentState) -> dict: draft = llm.invoke(f"Write a summary based on: {state['research']}") return {"draft": draft.content} def review_node(state: AgentState) -> dict: # Human-in-the-loop: graph pauses here, waits for approval return {} # Updated externally via graph.update_state() def route_after_review(state: AgentState) -> str: return END if state["approved"] else "write" # Loop back if rejected # 3. Build the graph workflow = StateGraph(AgentState) workflow.add_node("research", research_node) workflow.add_node("write", write_node) workflow.add_node("review", review_node) workflow.set_entry_point("research") workflow.add_edge("research", "write") workflow.add_edge("write", "review") workflow.add_conditional_edges("review", route_after_review) # 4. Compile with a checkpointer for persistence checkpointer = MemorySaver() graph = workflow.compile(checkpointer=checkpointer, interrupt_before=["review"]) # 5. Run it — the graph pauses at "review" waiting for human input config = {"configurable": {"thread_id": "session-001"}} result = graph.invoke( {"messages": ["Explain LangGraph for beginners"], "approved": False}, config=config )

The five capabilities that matter in production

Durable Execution

If your server restarts mid-workflow, LangGraph picks up exactly where it left off. Every transition is checkpointed. No lost progress, no re-running completed steps.

Human-in-the-Loop

interrupt_before=["approve"] pauses the graph at any node for human review. A compliance officer can inspect the agent's proposed action before it executes. Resume with graph.update_state().

Time-Travel Debugging

Every past state is stored. You can fork the graph from any checkpoint, modify the state, and replay from there. Debug production failures by rewinding to the moment before things went wrong.

Multi-Agent Supervision

Nodes can be other graphs. Build a supervisor that routes tasks to specialist sub-agents — Researcher, Writer, Critic. Each is a full StateGraph. The supervisor is the meta-graph.

Long-Term Memory

Short-term memory lives in the checkpointed state. Long-term memory persists across separate sessions in Postgres or a vector store. The agent remembers user preferences from last week.

The multi-agent supervisor pattern

from langgraph.graph import StateGraph, END # Each specialist is a compiled sub-graph researcher_graph = build_researcher_graph() # searches and synthesises writer_graph = build_writer_graph() # drafts content critic_graph = build_critic_graph() # reviews and scores def supervisor(state): # Decide which specialist to route to based on current state if not state["research_done"]: return {"next": "researcher"} elif not state["draft_done"]: return {"next": "writer"} elif state["score"] < 8: return {"next": "critic"} # Loop: revise until score >= 8 else: return {"next": END} multi_agent = StateGraph(SupervisorState) multi_agent.add_node("supervisor", supervisor) multi_agent.add_node("researcher", researcher_graph) multi_agent.add_node("writer", writer_graph) multi_agent.add_node("critic", critic_graph) multi_agent.set_entry_point("supervisor") # Conditional routing: supervisor decides next step multi_agent.add_conditional_edges("supervisor", lambda s: s["next"])

Why LangGraph matters for your career: Simple LangChain agents break at scale. LangGraph is what you need to build the kind of systems that Fortune 500 companies actually deploy — agents that survive failures, handle approvals, and coordinate multiple specialised sub-agents on complex tasks. If you want to build AI infrastructure, not just AI demos, LangGraph is the foundation.

How the three layers fit together.

The confusion most people have is treating these as alternatives. They're not. They form a stack, each at a different level of abstraction:

Layer 3
LangChain — High-level application framework. Provides chains, agents, memory, and 300+ integrations. The batteries-included API for building LLM apps quickly. Under the hood, LangChain agents now run on LangGraph.
Layer 2
LangGraph — Stateful orchestration runtime. The state machine that controls agent loops, branching, checkpointing, and multi-agent coordination. Use directly when you need full control. Use implicitly through LangChain when you don't.
Layer 1
FastMCP — Tool connectivity protocol. Defines and serves the tools your agents can call. Model-agnostic, framework-agnostic. One MCP server works with Claude Desktop, a LangGraph agent, or any other MCP client.

In a real production system, the three work like this:

  1. Your team builds a FastMCP server that exposes your internal tools — a CRM query tool, a ticket creation tool, a SQL runner. One server, reusable by any agent framework.
  2. A LangGraph state machine runs the agent loop — routing between an LLM reasoning node and a tool-execution node, checkpointing state at each step.
  3. LangChain provides the LLM interface, the prompt templates, and the MCP adapters that translate your FastMCP tools into LangChain-compatible tool objects.

The LangChain MCP Adapters library is the glue:

from langchain_mcp_adapters.client import MultiServerMCPClient from langgraph.prebuilt import create_react_agent from langchain_anthropic import ChatAnthropic # Connect to one or more FastMCP servers async with MultiServerMCPClient({ "crm": {"url": "http://crm-mcp-server:8000/sse"}, "ticketing": {"url": "http://tickets-mcp-server:8000/sse"}, }) as client: # MCP tools become first-class LangChain tools tools = await client.get_tools() # LangGraph ReAct agent with MCP tooling agent = create_react_agent( ChatAnthropic(model="claude-opus-4-8"), tools ) result = await agent.ainvoke({ "messages": [{"role": "user", "content": "Find all open support tickets for Acme Corp."}] })

When to pick which approach:

Scenario Use Why
Exposing your internal APIs to any AI tool FastMCP Framework-agnostic. Build once, use everywhere MCP is supported.
Quick chatbot, single-turn Q&A, simple RAG LangChain 300+ integrations, fast prototyping, clean LCEL syntax.
Long-running agent, approval workflows, multi-step tasks LangGraph Checkpointing, human-in-the-loop, durable execution.
Multi-agent: Researcher + Writer + Critic loop LangGraph + FastMCP Supervisor graph with MCP tools shared across sub-agents.
Full production system: tools + orchestration + apps All three FastMCP (tools) → LangGraph (runtime) → LangChain (API layer).

The learning path.

The order matters. If you start with LangGraph before you understand what LangChain is doing, you'll be fighting the abstraction levels. Here's the sequence that works:

Week 1–2: LangChain fundamentals. Understand what a chain is, how LCEL pipes components together, how a basic agent calls tools, and how RAG works. Build three things: a chatbot with memory, a document Q&A pipeline, and a tool-calling research agent. Don't skip this even if you're impatient to get to LangGraph — LangGraph will make much more sense once you've felt the limits of the linear agent loop.

Week 3–4: LangGraph. Start with the LangChain Academy course (free, official, 7 modules). Build a StateGraph with a conditional edge. Add a checkpointer and watch the graph survive a restart. Implement a human-in-the-loop approval step. Then build the multi-agent supervisor pattern — at that point, LangGraph clicks.

Week 5–6: FastMCP. Build a FastMCP server that wraps an API you use: GitHub, Notion, a REST endpoint you own at work. Connect it to Claude Desktop to see it work instantly. Then wire it into your LangGraph agent using LangChain MCP Adapters. The integration is the real skill — not just building servers, but building servers that slot cleanly into agent systems.

Ongoing: Build something real. The patterns that matter come from building, not reading. Suggestions: a personal research assistant that queries your notes and the web, an automated code review pipeline with a human approval gate, or a customer service agent that reads from your company's knowledge base and escalates to humans on uncertainty.
